# Skills
skills is basically folders of instructions . We require it when we perform certain repetative tasks like code review , style of code commit etc.
For personal skills , we can create a folder .claude in the root of the computer and create a folder with the name of skills and inside it create a file with the name of skill and name it as skills.md.
    For Project skills , we can create a folder .claude in the root of the project and create a folder with the name of skills and inside it create a file with the name of skill and name it as skills.md.



# Create a skill
```
---
name: pr-description
description: Writes pull request descriptions. Use when creating a PR, writing a PR, or when the user asks to summarize changes for a pull request.
 ---

When writing a PR description:

1. Run `git diff main...HEAD` to see all changes on this branch
2. Write a description following this format:

 ## What
One sentence explaining what this PR does.

 ##  Why
Brief context on why this change is needed

 ## Changes
- Bullet points of specific changes made
- Group related changes together
- Mention any files deleted or renamed
```


# Hierarchy of skills
Enterprise > Project > Personal > Plugins

To avoid conflicts use better description


# Skills meta fields
name (required) — Identifies your skill. Use lowercase letters, numbers, and hyphens only. Maximum 64 characters. Should match your directory name.

description (required) — Tells Claude when to use the skill. Maximum 1,024 characters. This is the most important field because Claude uses it for matching.

allowed-tools (optional) — Restricts which tools Claude can use when the skill is active. Example git , browser , linux commands , etc.

model (optional) — Specifies which Claude model to use for the skill
Ex : claude-3-5-sonnet

# Progressive Disclosure

scripts/ — Executable code
references/ — Additional documentation
assets/ — Images, templates, or other data files

# skills vs other claude code features

claude.md  --> loads into every conversation whereas skills load on demand when ask for a review.
 
 subagents --> run in a separate context , skills run in the same context.

 hooks --> run on specific events like file save , commit , push , etc.

 MCP --> provides external tools

 # Troubleshooting skills

- Agent skills verifer --> use 'validate' command to check if the skill is valid.
- check description --> add trigger phrases , "why it is slow" , "make it faster" . Use distinct description for each skill.
- claude --debug 
- Plugin skills arent appearing --> clear cache use rm and reinstall.
- runtime errors  --> add "Requires playright to function properly" to description.
